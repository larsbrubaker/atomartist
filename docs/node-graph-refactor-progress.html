The big change in one paragraph

The node-graph engine used to pin sockets to the NodeDef trait via &'static str keys, with edges identified by (NodeId, &'static str). That meant per-instance dynamic sockets, rename-without-edge-churn, and reorder were all blocked. We pivoted to instance-owned sockets with stable UIDs: every node now carries a Vec<Socket> for its inputs and outputs, each socket has a graph-unique SocketUid(u64), and noodles reference those uids exclusively. Names are display-only and can change freely without touching noodles. This unlocked the Blender-style dynamic-input Output node (one empty trailing slot, adopts source name+type on connect, appends a new empty slot, removes on disconnect, duplicate-source rejection) and the same shape now powers Combine.

Vocabulary — get the names right

Stored in ~/.claude/projects/…/memory/atomartist_vocabulary.md so future sessions stay consistent. The user is sensitive to inconsistency here.

Node
A box in the graph. Types: NodeId, NodeInstance, NodeDef, NodeRegistry.
Graph
The whole DAG (the document).
Socket
Input/output connector — the dot/pin on a node row. Carries SocketUid, name, display_label, socket_type, optional.
input / output
Direction is implied by which list a socket lives in: NodeInstance.inputs vs .outputs.
Noodle
The ribbon between two sockets. Types: Noodle, NoodleEndpoint. Never "edge", "wire", "link", or "connection" for this concept.
Property
An editable scalar on a node row. PropDef, NodeProperties, node.properties.

Mesh-side "edge" (triangle edges in geometry code) stays "edge". The rule applies only to the noodle concept.

What landed (Stages 1 & 2)

Stage 1 — engine refactor

  • New socket primitives: SocketUid(u64), Socket { uid, name, display_label, socket_type, optional }, SocketUidAlloc (in atomartist-lib/src/graph/socket.rs).
  • NodeInstance owns sockets: inputs: Vec<Socket>, outputs: Vec<Socket>, cached_outputs: HashMap<SocketUid, PortValue>. Properties keyed by Arc<str> for runtime-allocated names.
  • Noodles use uids: Noodle { from: NoodleEndpoint, to: NoodleEndpoint } where endpoint is (NodeId, SocketUid).
  • NodeDef trait shrank to factory + behavior: instantiate(alloc) -> InstanceTemplate, evaluate(ctx) -> NodeOutputs, plus optional hooks validate_input_connection, on_input_connected, on_input_disconnected.
  • Granular socket-mutation API on Graph: rename_socket, relabel_socket, retype_socket, append_input_socket, append_output_socket, remove_input_socket, remove_output_socket, reorder_input_sockets, noodles_touching. (Lives in graph/socket_mutations.rs to keep graph.rs under the 800-line cap.)
  • Serialization v2 (graph_json.rs): per-node socket lists, noodles by uid, next_socket_uid for allocator resume. v1 saves no longer load (sanctioned break).
  • ~25 existing nodes ported to instantiate() + uid-keyed EvalCtx.

Stage 1i — cleanup

  • Deleted nodes/io/graph_output_node.rs. The unified Output node plays both viewport-anchor and subgraph-output-declarator roles.
  • Rebuilt SubgraphNodeDef to scan templates for Output's mirror outputs (skipping the internal __display__) as the published subgraph ports.
  • Rebuilt CombineNode on the dynamic-input mechanism (second consumer that validates the design). Placeholder type is Geometry3d (constrained), not None (wildcard) — proves the design generalizes.

Stage 2 — unified Output node

atomartist-lib/src/nodes/output_node.rs — Blender-style dynamic multi-input. Trailing empty slot type None (wildcard, accepts any source). On connect: adopts source name + type, formats display label as "{NodeTitle} - {socket_name}", mints a mirror output socket, appends a fresh empty. validate_input_connection rejects duplicate-source. on_input_disconnected collapses the slot + removes the mirror. evaluate mirrors each named input to its named output, plus merges every Geometry3d input into a synthetic __display__ output the viewport reads.

Engine support: SocketType::None is now a wildcard target type (any source can land on it). Symmetric direction not allowed — outputs always carry a concrete type.

Stage 3 — wgpu rendering port (NodeDesigner parity)

Behavioural port of NodeDesigner's three-viewer.js chain into the Rust + wgpu pipeline. Source-level translation is impossible (different language, different API, naga's WebGL2 backend can't sample depth textures as f32); algorithms + constants are matched verbatim.

  • Blinn-Phong shader — view-space lighting via dpdx/dpdy flat normals, dual camera-fixed directional lights with independent diffuse + specular + per-light ambient, configurable shininess (default 30), sRGB→linear base colour. Defaults match NodeDesigner's createDepthPeelMaterial uniforms (light 0 = (-1,-1,1).normalize(), light 1 = (1,1,1).normalize(), diff 0.7 / 0.5, spec 0.05 / 0.05, global ambient 0.2). Shared Uniforms struct between the opaque pass and the dual-peel colour pass so peeled fragments shade identically. File: atomartist-renderer/src/scene_renderer/opaque_shaders.rs.
  • Post-process selection outline — port of selection-outline.js: ID prepass rasterises selected mesh into R8 mask + R32Float depth mirror, edge-detect quad samples 8 neighbours + full-scene depth, alpha-blends a rim at colour.a=1.0 (visible) or occluded_alpha=0.35 (behind geometry). Runs inside the per-sample HDR stream so the rim is AA-smoothed by the Halton accumulator. The legacy inverted-hull outline is gone. Colour matches NodeDesigner blue (0x88ccff dark / 0x4477cc light). File: scene_renderer/post_outline/.
  • Gizmo render pass — port of gizmo-render-pass.js as two-variant LineList pipelines (solid depth-tested against scene_depth + overlay no-depth alpha-blended at occluded_alpha). Targets the per-sample HDR view so gizmos AA-smooth with the rest of the scene. WgpuSceneRenderer.gizmo_lines: Vec<GizmoLineSet> populated each frame by the host. File: scene_renderer/gizmo_pass/.
  • Bounds gizmo — 12 edges around the selected mesh's AABB, red 0xff4444, occluded_alpha 0.25, built by GizmoLineSet::bounds_box(center, size, matrix). Host pushes it from viewport_widget.rs whenever selection is active.
  • Verified byte-for-byte ports: accumulation AA (Halton(2,3), MAX_SAMPLES=16, mix-weight 1/(N+1), Rgba16Float ping-pong) and contact shadows (1024² shadow map, ±100 bounds, blur radius 4.0, opacity 0.35) already match NodeDesigner exactly — no code change needed.
  • WebGL2 / naga guardrails: scene_shaders_emit_glsl_es_300, peel_shaders_emit_glsl_es_300, outline_shaders_emit_glsl_es_300, gizmo_shader_emits_glsl_es_300 — each runs the WGSL through naga's GLSL ES 3.00 backend and asserts no sampler2DShadow sentinel. Catches future shader changes that drift into a path naga can't emit cleanly.

Stage 4 — per-node matrix + colour data model

Every geometry-producing node now carries a matrix + color property (mirrors NodeDesigner's socket._data.matrix / _data.color), so handles + control gizmos have somewhere to write transform updates back into. Mechanism:

  • atomartist_lib::geometry::Geometry3d { mesh: Arc<MeshGL>, matrix: [f32; 16], color: [f32; 4] } — the bundle that travels along a Geometry3d socket. PortValue::Geometry3d now holds Arc<Geometry3d>.
  • registry::geometry_props() returns the standard [matrix=identity, color=DEFAULT_GEOMETRY_COLOR] PropDefs every geometry-producing node folds into its properties().
  • registry::wrap_mesh(ctx, mesh) reads those props off ctx and builds a Geometry3d ready to wrap in PortValue::Geometry3d.
  • All 16 geometry-producing nodes updated (every primitive + Transform, Align, FitToBounds, Boolean, Combine, Extrude, LibraryMesh, MeshNode, Output).
Known semantic gap (deferred): ops nodes (Transform, Align, FitToBounds, Boolean, Combine) bake their internal transform into the mesh vertices, then emit a Geometry3d whose matrix is the node's own gizmo-driven matrix (identity by default). The upstream's matrix is currently dropped on the floor by these ops — they call apply_transform(&input.mesh, …) without composing input.matrix. Output node compensates by baking each upstream's matrix into the merged display mesh as a one-off, but Transform → Transform chains still drop the upstream matrix. Either compose matrices through ops, or treat the matrix as render-time only (don't bake). Decide before the control gizmos ship.

Next up — ordered TODO priority order

This is the queue. Top is highest priority. Items below the divider are smaller / independent and can be picked up out of order.

1. Handle meshes (drag-on-line / drag-on-plane) next

  • JS source: NodeDesigner/static/js/node-editor/rendering/handle-meshes.js (257 lines).
  • Purpose: per-vertex draggable handles for "drag-on-line" / "drag-on-plane" nodes — user clicks a small sphere/cube in 3-D and drags it along an axis or plane to drive a node property.
  • Rendering side is unblocked: emit handle geometry as a GizmoLineSet (or small filled-tri variant — see note below) and the existing gizmo pass draws it.
  • What's missing:
    1. Mouse-raycast against handle geometry (extend picking.rs with a sphere/AABB intersect for handles).
    2. Drag state machine on Viewport3dWidget — mouse-down on handle → record start property value → mouse-move updates the property via a coalesced undo command → mouse-up commits.
    3. Property write must go through an UndoCommand so the canvas right-side numeric editor stays in sync.
  • The gizmo pipeline today is LineList only. Handles want filled spheres/cubes. Decision needed: either add a filled-tri pipeline variant to gizmo_pass, or extend GizmoLineSet with a topology enum.

2. Control gizmos (Z, XY, rotate-corner) queue

  • JS sources (each is a pair):
    • z-control-gizmo.js (542 lines) + z-height-drag.js (1204 lines)
    • xy-control-gizmo.js (561 lines) + xy-width-depth-drag.js (807 lines)
    • rotate-corner-gizmo.js (813 lines) + rotate-corner-drag.js (274 lines)
  • Each gizmo:
    • Mounts above the selected mesh; geometry mostly line segments + a few arrowhead/box markers.
    • Shows constant on-screen size (must scale with camera distance per paint — see NodeDesigner's per-frame updateZControlPosition etc.).
    • Drag updates the node's matrix property (Stage 4's new property), which propagates downstream via the new Geometry3d bundling.
  • Z control: vertical translate arrow + height-scale box anchored at base. XY control: 4 edge arrows constraining each axis. Rotate-corner: ring + corner arrows.
  • Each share the same picking + drag-state pattern as #1; once #1 lands, these reuse it.
  • Decide first: matrix composition through ops nodes (see the yellow box at the end of Stage 4). Until that's resolved, dragging a gizmo upstream of a Transform node will give surprising results.

3. Bounds gizmo edit handles extension

  • Bounds gizmo currently renders read-only (Stage 3 / Task #9 shipped). NodeDesigner allows dragging the 8 corner / 6 face handles to resize the bounds (for FitToBounds nodes).
  • Same picking + drag-state pattern as #1 + #2; writes FitToBounds.width / .height / .depth properties.

4. Measurement overlay substantial

  • JS: measurement-overlay.js (635 lines).
  • Renders dimension lines + tick marks + value labels around selection. Mixed 3-D (the line segments, depth-tested against geometry) + 2-D (the value labels, screen-space text).
  • Lines side: reuses the gizmo pass.
  • Labels side: need agg-gui text overlay anchored to world points → projected to screen each paint.

5. Rubber-band selection visual small

  • JS: rubber-band-selection.js (210 lines).
  • Pure 2-D screen-space rectangle drawn over the viewport during drag-select. No depth interaction; lives in atomartist-ui, not the renderer.
  • Hit-test logic (which meshes fall inside the rect) already lives in picking.rs — just need the visual.

6. Lighting debug panel UI-only

  • JS: lighting-panel.js (434 lines) + lighting-debug-panel.js (361 lines).
  • All the uniforms it tweaks are already wired: WgpuSceneRenderer.light_diffuse0 / light_specular0 / light_ambient0 / light_diffuse1 / light_specular1 / global_ambient / material_specular / shininess. Cache fingerprint already invalidates accumulation on tweaks.
  • Missing: an atomartist-ui floating panel with sliders + colour pickers bound to those fields. Pure UI work.

7. Vertex format expansion larger

  • NodeDesigner's depth-peel shader supports per-vertex colour, per-vertex alpha, barycentric (for the wireframe overlay shader), and UV (for textured materials). atomartist's Vertex struct is pos[3] + normal[3] only.
  • Touches: opaque_pass::Vertex, vertex_layout, mesh ingestion in WgpuSceneRenderer::ensure_mesh_buffers (synthesise barycentric per-triangle since MeshGL doesn't carry it), shader logic in both SCENE_SHADER + DUAL_PEEL_COLOR_SHADER, plus new useVertexColors / useVertexAlpha / showWireframe uniforms.
  • Defer until a node actually emits per-vertex colour data (manifold-rust's MeshGL.num_prop > 6 case isn't wired in any of our nodes yet).

Open threads (lighter / engine-side gaps) deferred

  • NodeDesigner test ports — 2 files queued. graph-io-nodes.test.ts + graph-io-slot-rearrangement.test.ts and socket-rename-persistence.test.ts. Pattern matches the five already-ported files; shared fixtures in tests/common/mod.rs.

  • Multi-mesh viewport. Output node merges all geometry inputs into one mesh on __display__. For per-input materials / per-input picking, extend last_mesh_output to Vec<Arc<Geometry3d>> (was Arc<MeshGL> pre-Stage-4).
  • File-line guardrail. demo-native/src/main.rs is at 815 lines (over the 800-line cap) as of this snapshot. Not touched by recent work; the first_party_project_files_stay_under_line_limit test is the canonical reporter. Refactor or carve a sub-module when convenient.
