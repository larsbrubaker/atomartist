AtomArtist — what's left to reach NodeDesigner / MatterCAD parity

Outstanding-work catalogue. Reference sources at C:\Development\FDS\NodeDesigner (JS / TS) and C:\Development\MatterCAD (C#).

Snapshot: Engine, multi-body Geometry3d, dual depth-peel, contact shadows, accumulation AA, selection outline (single-body), Stage-4 per-node matrix + colour, dynamic-input Output node, and ~25 node types are landed. The remaining work is the bulk of what makes the application feel like NodeDesigner / MatterCAD — the long tail of node types, property editors, gizmos, editing UX, and file I/O. This document enumerates that tail.

Vocabulary

Node
Box in the graph. Rust types: NodeId, NodeInstance, NodeDef, NodeRegistry.
Graph
The whole DAG (the document).
Socket
Input/output pin. Carries SocketUid, name, display_label, socket_type, optional.
Noodle
Ribbon between two sockets. Never "edge", "wire", "link", or "connection".
Property
Editable scalar on a node row. PropDef, NodeProperties.
Body
One renderable mesh inside a Geometry3d — carries its own mesh, matrix, colour, and optional per-vertex colour buffer. Multi-body groups iterate per draw pass.

Mesh-side "edge" (triangle edges) stays "edge". Rule applies to the noodle concept only.

1. Node coverage — the long tail

Current AtomArtist: 8 3-D primitives, 4 2-D primitives, 5 3-D ops, 3 2-D ops, 2 math, 1 mesh-import (~25 total). MatterCAD ships ~80 Object3D subclasses; NodeDesigner ~40. Below is the gap, grouped so a sitting can pick one cluster and finish it.

1.1 Missing 3-D primitives small

  • Half-Sphere — MatterCADLib/DesignTools/Objects3D/Primitives3D/HalfSphereObject3D.cs. Half a UV sphere, planar cap. Reuse generate_sphere with a v-range clip + cap fan.
  • Half-Cylinder — .../HalfCylinderObject3D.cs. Reuse generate_cylinder_advanced with start_angle=0, end_angle=π (already supported by the partial-revolve path — just expose as a node).
  • Half-Wedge — .../HalfWedgeObject3D.cs. Custom wedge variant.
  • Ring (flat 3-D) — .../RingObject3D.cs. Already have Ring in 2-D primitives; the 3-D version is a thin extruded annulus.
  • Tube — NodeDesigner extended primitive. Cylinder shell.

1.2 Missing transform / placement ops small-medium

  • Mirror — .../Operations/MirrorObject3D_2.cs. Reflect about XY / YZ / XZ plane. Negative-determinant transform — winding-flip code already exists in geometry/mesh3d.rs::apply_transform (see the bug note in the "Decisions pending" section below if you haven't already swapped tri order on det < 0).
  • Array Linear — .../Operations/ArrayObject3D_2.cs. Count + spacing on each axis. Each instance becomes its own Body in the output multi-body group — the renderer already iterates bodies.
  • Array Radial / Polar — implied by MatterCAD's array family. N copies around an axis.
  • Set Center — .../Operations/SetCenterObject3D.cs. Shifts geometry so a named anchor (min/center/max on each axis) lands at the origin.
  • Rotate-only / Scale-only / Translate-only — MatterCAD splits these from the combined matrix node (Operations/TranslateObject3D.cs, RotateObject3D_2.cs, ScaleObject3D_3.cs). Convenience nodes; data path = current Transform.

1.3 Missing deformations + mesh ops medium-heavy

  • Twist — .../Operations/TwistObject3D.cs. Linear angular twist along an axis. Vertex-shader-friendly (per-vert rotation) or CPU pre-bake.
  • Pinch (v3) — .../Operations/PinchObject3D_3.cs. Lerp vertices toward an axis.
  • Radial Pinch — .../Operations/RadialPinchObject3D.cs.
  • Bevel (edge / corner) — .../Operations/BevelObject3D.cs. Geometry-heavy: detect edges by dihedral angle, split + chamfer.
  • Decimate — .../Operations/DecimateObject3D.cs. Vertex-cluster decimation; manifold-rust may already expose this — check before reimplementing.
  • Repair — .../Operations/RepairObject3D_2.cs. Close gaps, weld vertices, drop tiny triangles. Likely belongs in manifold-rust as a general op (note in CLAUDE.md says to push upstream when it generalises).
  • Hollow Out — .../Operations/HollowOutObject3D.cs. Shell offset minus original. Needs an inner-offset op in manifold-rust.
  • Plane Cut — .../Operations/PlaneCutObject3D.cs. Slice along a plane, cap the cut. Manifold-rust supports this — wire up a node.
  • Dual Contouring — .../Objects3D/DualContouringObject3D.cs. Iso-surface meshing from a field. Pair with the field-system node work below.
  • Revolve — .../Operations/Path/RevolveObject3D.cs. 2-D path → solid of revolution. Lathe op.
  • Find Slice — .../Operations/FindSliceObject3D.cs. Extract a planar cross-section; output is a 2-D path.

1.4 Path / 2-D pipeline expansion medium

  • Border Path — .../Operations/Path/BorderPathObject3D_2.cs.
  • Outline Path — .../Operations/Path/OutlinePathObject3D.cs. Distinct from our existing Stroke — outline produces a closed boundary around a shape.

  • Image (texture / heightmap source) — .../Objects3D/ImageObject3D.cs. Wraps an image asset as a node output.

  • Gear — .../Mechanical/GearObject3D.cs. Involute gear profile generator.

1.6 Structural / hierarchy nodes requires graph extension

  • Group / Ungroup — .../Operations/GroupObject3D.cs. Multi-body passthrough that's editable as a unit on the canvas. Renderer already iterates bodies; the editing UX is the work.
  • Component — .../Objects3D/ComponentObject3D.cs. A reusable subgraph saved to disk and dropped in by reference. Our existing SubgraphNode covers the in-document case; component-as-file needs an asset-ref + reload pipeline.
  • Selection Group — .../Objects3D/SelectionGroupObject3D.cs. Selects a subset of bodies from a multi-body input. Pairs with the body-picker work flagged in §3.
  • Description — .../Objects3D/DescriptionObject3D.cs. Comment / annotation node, no geometry output.
  • Measure Tool — .../Objects3D/MeasureToolObject3D.cs. Outputs distance / angle scalars from picked points. Pairs with the measurement-overlay viewport work.

1.7 Field / Blender-style per-pixel evaluation big architectural

NodeDesigner has a parallel evaluation model (static/js/node-editor/.../field-system.js): some node outputs are fields (functions of position) rather than baked values. Downstream nodes sample the field per-vertex or per-pixel. Powers ColorInput, PositionInput, MathField, MixField, FormulaField, and feeds into dual contouring for SDF-style modelling.

  • Decision: do we add a new PortValue::Field variant carrying a closure/expression, or treat fields as a separate evaluation kernel that runs at consumer time? NodeDesigner does the latter — sample on demand.
  • Body picker (Blender-style selector from a multi-body group) is the smaller cousin of this — explicitly requested by the user when we agreed multi-body should support both per-body and per-vertex colour.
  • Pairs with Dual Contouring (§1.3) and the colour-system node cluster in NodeDesigner (nodes-color.js).

3. Multi-body — outstanding pieces

Multi-body opaque + dual-peel + shadow caster + per-vertex colour all landed. The remaining items are below.

  • Multi-body selection outline — outline pass still single-body (phase-1 fallback in render_impl.rs). Add body BGL binding to the outline ID pipeline, loop bodies in post_outline::execute. Outline doesn't show colour so this is purely a silhouette extension.
  • 3MF importer emits multi-body — current mesh-drop path flattens to a single body. 3MF supports per-body materials + colours; the importer (file format support, see §6) should fan-out into multiple Body entries with their assigned matrix + colour. Source pattern at MatterCADLib/.../ThreeMfImporter.cs.
  • Body picker / filter nodes — Blender-style selector that takes a multi-body group and yields a subset. Field-style predicates over body indices, names, tags, bounding regions. Pairs with the SelectionGroup node (§1.6).
  • Per-vertex colour producers — the renderer's per-vertex colour shader path is in place but no current node emits per-vertex data. First consumer will be the boolean ops once they're upgraded to propagate colour through the cut surface (each output vertex inherits from whichever input it came from).
  • Per-body picking — picking.rs currently returns mesh-level hits. Extend to (body_index, tri_index) so the UI can highlight a single body inside a multi-body group.

4. Viewport / 3-D interaction

4.1 Control gizmos (mostly missing) heavy

Bounds gizmo renders read-only. The four NodeDesigner / MatterCAD control gizmos that let users drag transforms in 3-D are not built. Each needs picking + drag-state + property-write-through-undo.

  • Z control — rendering/z-control-gizmo.js (542 lines) + z-height-drag.js (1204 lines). Translate-Z arrow + height-scale.
  • XY control — xy-control-gizmo.js (561 lines) + xy-width-depth-drag.js (807 lines). 4 edge arrows.
  • Rotate corner — rotate-corner-gizmo.js (813 lines) + rotate-corner-drag.js (274 lines). Corner-mounted rotation handles.
  • Bounds edit handles — extend the read-only bounds gizmo with 8 corner + 6 face drag handles for resizing FitToBounds.
  • Handle meshes — handle-meshes.js (257 lines). Per-vertex draggable spheres for "drag-on-line" / "drag-on-plane" nodes. The picking + drag state pattern from this lands first, then the four gizmos above reuse it.
  • Filled-tri gizmo pipeline — current gizmo_pass is LineList only. Handles want filled spheres/cubes — either add a filled-tri variant or extend GizmoLineSet with a topology enum.

4.2 Overlays + indicators

  • Measurement overlay — measurement-overlay.js (635 lines). Dimension lines + tick marks + value labels around the selection. Lines reuse the gizmo pass; labels need agg-gui text overlay anchored to world points → projected to screen each paint.
  • Axis indicator — .../SceneViewer/{AxisIndicator,ScreenspaceAxisIndicator}Drawable.cs. Small XYZ axis cluster in a viewport corner.
  • Rubber-band selection visual — rubber-band-selection.js (210 lines). 2-D screen-space rectangle during drag-select. Hit-test logic already in picking.rs; just the visual is missing.
  • Section / cutaway plane — MatterCAD's plane-cut visualisation. Render the cut surface with a stippled pattern. Build on the PlaneCut node (§1.3) once it lands.
  • Normals debug — .../SceneViewer/NormalsDrawable.cs. Toggle vertex-normal arrows. Useful while we still have winding gremlins.

4.3 Camera + render style

  • Orthographic toggle — MatterCAD + NodeDesigner both expose perspective ↔ ortho. OrbitCamera::projection_matrix currently only emits perspective.
  • Hidden-line render style — .../View3D/ModelRenderStyle.cs lists shaded / wireframe / hidden-line. We have the first two; hidden-line draws only silhouette + crease edges.
  • Lighting debug panel — lighting-panel.js (434) + lighting-debug-panel.js (361). All uniforms it would tweak are already wired on WgpuSceneRenderer; missing piece is the atomartist-ui floating panel.
  • Vertex format expansion — NodeDesigner's shader supports UV (textured materials) and barycentric (wireframe-overlay shader). Our Vertex is pos+normal+(new)color; add UV when a textured material node lands.

5. Editing UX — the big chunk that makes it feel like an editor

5.1 Selection + manipulation

  • Copy / cut / paste / duplicate — NodeDesigner's canvas.copyToClipboard / pasteFromClipboard + ui/edit-menu-actions.js. We have deletion; clipboard ops are missing. Tricky bits: socket UIDs must regenerate on paste, noodles inside the pasted subgraph must remap.
  • Group / ungroup — collapse selection to a GroupObject3D-style container, expand back. Pairs with §1.6.
  • Multi-select drag — engine supports selection; canvas-side multi-drag with consistent offset is missing.
  • Alignment / distribution tools — MatterCAD has Align-Left / Center-Horiz / Distribute-Vertical etc. as menu actions on the selected nodes (or selected 3-D objects). Map both layers.
  • Snap to grid / smart snap guides — graph/state/smart-snap.js. PowerPoint-style alignment + spacing guides during node drag. Threshold 8px canvas-space. agg-gui has window snapping (in flight separately); the node-canvas variant is a port of that pattern.

5.2 Undo / redo stack expansion

Engine has ChangeTracker (JSON snapshot dirty bit). For a real editor we need a typed undo command stack matching MatterCAD's pattern (PartPreviewWindow/View3D/UndoCommands/):

  • InsertCommand / DeleteCommand (already partially landed as graph mutations; not yet wrapped in undo).
  • TransformCommand — coalesces drag-during-frame writes.
  • ChangeColor, ChangeOpacity, SetOutputType, MakeHole.
  • UngroupCommand.
  • Inline-edit (rename, property edit) coalescing — NodeDesigner's undo-buffer.js (264) is the reference. Each property editor coalesces same-property writes within a debounce window.

5.3 Node-canvas affordances medium

  • Node search palette — core/node-search.js. Fuzzy-match by title + type, grouped by category, sortable by relevance. Triggered by Tab or a "+" button.
  • Inline rename — node title + socket name. Graph::rename_socket exists; canvas-side inline-edit widget is not built.
  • Drag-reorder input sockets — engine ready (Graph::reorder_input_sockets); canvas-side row-drag handler is not.
  • Right-click context menu expansion — current menu has Delete + Add. NodeDesigner / MatterCAD versions cover Duplicate, Group, Make Component, Change Colour, Set Output Type (Solid / Hole), Disable, Mute, Bypass, Comment, Frame.
  • Comment / frame nodes — visual-only nodes that group + caption a region of the canvas. NodeDesigner has it; we don't.
  • Mute / bypass — disable a node temporarily without disconnecting (passes input through).
  • Disconnect by drag — currently shipped; verify the bezier orientation fix holds at high zoom-out.

6. File I/O

  • Import: 3MF multi-body — .../ThreeMfImporter.cs + NodeDesigner's @jscadui/3mf-export usage. Use threemf-rs or write a minimal reader against the zip + XML spec. Each <object> + <item> becomes a Body; <basematerials> drives per-body colour.
  • Import: SVG — paired with §1.5 SVG node. usvg parses; kurbo flattens beziers.
  • Import: glTF — NodeDesigner supports via three.js. AtomArtist: gltf crate; useful for downstream BIM / archviz workflows.
  • Import: AMF, DXF — MatterCAD covers via separate libraries. Lower priority — most users hit STL / OBJ / 3MF.
  • Export pipeline — currently nothing user-visible. Needs:
    • Export node (writes from a sink). Or menu action "Export Output node's geometry to..." picker.
    • Format writers: STL (binary + ASCII), OBJ (with materials.mtl sidecar), 3MF (with per-body materials), SVG (for 2-D paths), glTF.
  • Native AtomArtist project format — currently graph_json.rs v2. Add:
    • File extension + magic header (e.g. .atmr + JSON-with-front-matter, OR a zip container per MatterCAD's .mcx).
    • Asset embedding (dropped meshes live in assets/ with sha256 ref — needs a zip container, not raw JSON).
    • Format-version field + migration ladder (we already broke v1; future bumps should migrate, not break).
    • Autosave + recent-files list — atomartist-ui responsibility.

7. Graph engine extensions

  • Async evaluation — matter-graph-async-execution.js (313 lines). Heavy ops (booleans, fit-to-bounds, mesh repair) should run off the UI thread. Re-eval debouncing + cancellation when a parameter changes mid-run.
  • Expression evaluation in number fields — see §2 (the *OrExpression editors). Cross-cuts engine + property panel: properties become formulas referencing other properties by name ($radius * 2 + 0.5). Sandboxed evaluator with variable lookup against the node's own properties + scene globals.
  • Drivers / animated values — NodeDesigner has time-driven node parameters. Lower priority unless animation export is on the roadmap.
  • Source-side hooks — engine has on_input_connected / _disconnected / validate_input_connection. Symmetric output-side hooks not implemented. Add when a node needs to react to its consumers.
  • Source-side wildcards — JS treats output type * as "match any input". atomartist supports target-side SocketType::None wildcard only.
  • Invalidation flags — MatterCAD's InvalidateType enum (Children, Color, Image, Matrix, Mesh, Name, OutputType, Path, Properties) is finer-grained than our dirty-bit. Important for performance once node counts grow.
  • Node tags / output type — MatterCAD's PrintOutputTypes { Default, Solid, Hole } drives boolean intent. Even without printing, "this geometry is a subtractor" tagging is useful as a semantic shortcut.
  • connectByType / connectByTypeOutput — JS convenience helpers flagged in tests/node_connections.rs. Add when a UI flow demands them.

9. Decisions still pending

  • Matrix composition through ops nodes. Ops like Transform, Align, FitToBounds, Boolean, Combine currently bake their internal transform into mesh vertices, then emit a Geometry3d whose matrix is the node's own gizmo matrix (identity by default). The upstream's matrix is dropped on the floor. Output node compensates by baking each upstream's matrix into the merged display mesh, but Transform → Transform chains still lose the upstream matrix. Decide before §4.1 ships: either propagate upstream matrices through every op, or commit to render-time-only matrices and strip baking everywhere. Without consistency, dragging a Z control on a box that feeds a Transform node behaves inconsistently.
  • Negative-determinant transforms. Mirror node (§1.2) needs apply_transform to flip triangle winding when det(M) < 0. Add a winding-flip step in geometry/mesh3d::apply_transform guarded on the sign of the 3×3 determinant.
  • Project file container. Raw JSON (current) vs zip-with-assets (MatterCAD's .mcx approach). Once dropped meshes / images need to round-trip with the save file, raw JSON breaks.
  • Field architecture. Closure-on-PortValue vs separate sample-time kernel (§1.7). Defer until the first field-consumer node lands.
  • Y-up world vs Z-up world. Geometry primitives now use Z-up cylinder axis (matching MatterCAD); world camera still Y-up. Either move world to Z-up everywhere, or accept the convention split and document it.
