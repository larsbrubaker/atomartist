AtomArtist — remaining work toward NodeDesigner / MatterCAD parity

Outstanding-work catalogue. Completed clusters are intentionally removed so this stays useful as a pickup list. Reference sources at C:\Development\MatterHackers\FDS\NodeDesigner (JS / TS) and C:\Development\MatterCAD (C#).

Snapshot: Engine, multi-body Geometry3d with per-vertex colour, dual depth-peel + contact shadows + accumulation AA (all multi-body), selection outline for the active body, matrix composition through ops (no more bake), Z-up world everywhere, per-body click selection via Body.origin NodeId, gizmo foundation (filled-tri pipeline + sphere/cube handles + ray-vs-AABB handle picking), XY bed-plane body drag + Z control drag with per-stroke matrix coalescing, dynamic-input Output node, ~26 node types, undo / redo stack wired through current canvas + viewport mutations, node snap guides, orthographic projection, shaded / wireframe render style, native .atmr project save/load with asset embedding, STL export, and mesh import via dropped files. Remaining work is the long tail that makes the application feel like NodeDesigner / MatterCAD — node types, richer property editors, remaining gizmo variants (rotate / bounds-edit / handle-meshes), higher-level editing UX, and broader file I/O.
Undo contract: every new mutation path lands with its matching UndoRedoCommand push. Use UndoBuffer::try_coalesce_last for in-stroke merging (slider / drag / typing). New action with no command pushed is a bug — see §5.2.

Vocabulary

Node
Box in the graph. Rust types: NodeId, NodeInstance, NodeDef, NodeRegistry.
Graph
The whole DAG (the document).
Socket
Input/output pin. Carries SocketUid, name, display_label, socket_type, optional.
Noodle
Ribbon between two sockets. Never "edge", "wire", "link", or "connection".
Property
Editable scalar on a node row. PropDef, NodeProperties.
Body
One renderable mesh inside a Geometry3d — carries its own mesh, matrix, colour, and optional per-vertex colour buffer. Multi-body groups iterate per draw pass.

Mesh-side "edge" (triangle edges) stays "edge". Rule applies to the noodle concept only.

1.6 Components / hierarchy requires graph extension

  • NodeDesigner-style Components — implement NodeDesigner's component model, not MatterCAD's GroupObject3D / ComponentObject3D object model. A component is a reusable subgraph stored inside the current .atmr as its own JSON document (NodeDesigner-style separate graph document), with component instances in the parent graph. The existing SubgraphNode covers part of the evaluation shape; remaining work is persistence, instance metadata, edit navigation, and UI affordances.
  • Full NodeDesigner input / output socket model — components require robust parity with NodeDesigner's graph boundary sockets, not just the current minimal GraphInput + Output bridge. Component definitions need stable input / output socket identity, names, display labels, type propagation, reorder / rename persistence, dynamic output adoption, and correct noodle remapping when a selection is extracted into a component. This should be implemented as a first-class component-port contract before the Save Selection as Component workflow ships.
  • Component drill-down editor — NodeDesigner opens component graphs in new tabs. AtomArtist should feel more like folder navigation: entering a component replaces the canvas with that component graph, and a breadcrumb bar shows the path back to the top-level graph. Breadcrumb segments should be clickable so users understand where they are and can climb back up without losing context.
  • Save selection as Component — context-menu action matching the intended NodeDesigner workflow: selected nodes are extracted into a new embedded component graph JSON, parent graph receives a component instance, internal noodles are preserved, and external noodles become boundary ports through the full NodeDesigner-style component input / output socket model.
  • Description — .../Objects3D/DescriptionObject3D.cs. Comment / annotation node, no geometry output.
  • Measure Tool — .../Objects3D/MeasureToolObject3D.cs. Outputs distance / angle scalars from picked points. Pairs with the measurement-overlay viewport work.

1.7 Field / Blender-style per-pixel evaluation big architectural

NodeDesigner has a parallel evaluation model (static/js/node-editor/.../field-system.js): some node outputs are fields (functions of position) rather than baked values. Downstream nodes sample the field per-vertex or per-pixel. Powers ColorInput, PositionInput, MathField, MixField, FormulaField, and feeds into dual contouring for SDF-style modelling.

  • Decision: do we add a new PortValue::Field variant carrying a closure/expression, or treat fields as a separate evaluation kernel that runs at consumer time? NodeDesigner does the latter — sample on demand.
  • Body picker (Blender-style selector from a multi-body group) is the smaller cousin of this — explicitly requested by the user when we agreed multi-body should support both per-body and per-vertex colour.
  • Pairs with Dual Contouring (§1.3) and the colour-system node cluster in NodeDesigner (nodes-color.js).

3. Multi-body — outstanding pieces

Multi-body opaque + dual-peel + shadow caster + per-vertex colour shader path + per-body click selection all landed. Remaining items below.

  • Multi-body selection outline for multi-select — the outline now targets the selected body's Body.origin instead of blindly falling back to body 0. Future multi-select / group-select should outline every selected body, which means the outline pass needs to iterate multiple selected body handles instead of a single outline_body_index.
  • 3MF importer emits multi-body — current mesh-drop path flattens to a single body. 3MF supports per-body materials + colours; the importer (file format support, see §6) should fan-out into multiple Body entries with their assigned matrix + colour. Source pattern at MatterCADLib/.../ThreeMfImporter.cs.
  • Body picker / filter nodes — Blender-style selector that takes a multi-body group and yields a subset. Field-style predicates over body indices, Body.origin NodeId tags, names, bounding regions. Pairs with the SelectionGroup node (§1.6).
  • Per-vertex colour producers — the renderer's per-vertex colour shader path is in place but no current node emits per-vertex data. First consumer will be the boolean ops once they're upgraded to propagate colour through the cut surface (each output vertex inherits from whichever input it came from).
  • Triangle-level picking — picking::pick_origin already returns the closest-body origin: NodeId by ray-vs-mesh against every body. Extend to also return the hit triangle index when a future use case (e.g. paint-on-face, vertex-snap) needs sub-body precision.

4. Viewport / 3-D interaction

4.1 Control gizmos — remaining variants extend per gizmo

Drag infrastructure landed: ray construction, body / handle picking, bed-plane projection, Z-axis projection, matrix write-back through AppState::set_node_matrix_with_undo with per-stroke coalescing. Body XY drag (mouse-down on a body's mesh → translate on the bed plane) and Z control drag (mouse-down on the sphere handle → translate along world Z) are wired. The pattern lives in viewport_widget/body_drag.rs (pure math) + viewport_widget/viewport_widget_interactions.rs (state machine). Remaining gizmos:

  • XY edge-arrow control — xy-control-gizmo.js (561) + xy-width-depth-drag.js (807). 4 edge arrows that scale width / depth (not whole-body translate — that's already covered by the bed-plane drag landed above). Reuse the existing handle-pick + drag-state pattern; the write target is the upstream primitive's width / depth property instead of matrix.
  • Rotate corner — rotate-corner-gizmo.js (813) + rotate-corner-drag.js (274). Corner-mounted rotation handles. Pattern: project ray onto the rotation plane → compute angle delta → compose into the body's matrix rotation.
  • Bounds edit handles — extend the read-only bounds gizmo with 8 corner + 6 face drag handles for resizing FitToBounds (writes width / height / depth).
  • Drag-on-line / drag-on-plane handles — handle-meshes.js (257). Per-vertex draggable spheres for node properties that expose a "grab this point" affordance. Geometry side already lands via sphere_handle / cube_handle; needs node-property → handle position binding + drag-back-into-property write.
  • Selection-while-dragging — currently a body XY drag promotes the dragged body to the active selection on mouse-up. NodeDesigner also shows the gizmos LIVE as soon as the body becomes selected mid-drag (i.e. on the first 2-px movement past threshold). Consider promoting selection earlier so the Z-control + future gizmos appear without waiting for mouse-up.

5.1 Selection + manipulation

This is the hierarchy workflow to implement instead of MatterCAD-style group / ungroup.
  • Enter / exit component navigation — double-click or context-menu enter should drill into a component graph; breadcrumbs replace NodeDesigner's tabbed component editing and provide the way back to the parent / top-level graph.

  • Alignment / distribution tools — MatterCAD has Align-Left / Center-Horiz / Distribute-Vertical etc. as menu actions on the selected nodes (or selected 3-D objects). Map both layers.

5.2 Undo / redo — remaining coverage extend per new action

Stack + bridge wired through agg-gui's UndoBuffer. Every AppStateModel mutation (add_node, remove_node, set_node_position, try_add_noodle, remove_noodle, set_property) plus AppState::import_mesh_file push commands. Slider + node drag coalesce per stroke via UndoBuffer::try_coalesce_last. Ctrl+Z / Ctrl+Y bound on the Edit menu. Outstanding:

  • New mutation paths land with their command. Future rotate / bounds-edit / handle-mesh gizmos (§4.1), context-menu Duplicate / Save as Component / Mute / Bypass (§5.3), inline rename (§5.3), drag-reorder sockets (§5.3), selection-marquee actions (§5.1), and file-load itself (currently clears history — could capture pre-load state for "Undo Load"). Each new action must push a matching *Cmd or coalesce into an existing one — leave nothing un-undoable.
  • Per-node-type ChangeColor / ChangeOpacity / SetOutputType coverage. ChangePropertyCmd works for any PortValue, but the Output Type tag (Default / Solid / Hole — MatterCAD pattern) doesn't exist as a property yet. When that lands (§7 "Node tags / output type"), confirm the set-output-type UI write goes through the same command.
  • Output node socket-hook side effects. Connecting to the Output node's trailing slot mutates the node's own input / output socket list inside the on_input_connected hook. Currently captured by the parent ConnectCmd via g.disconnect(...) in undo (which replays the on_input_disconnected side effect). Add a regression test that adds → undos a connect to Output and asserts the slot list shrinks back to "one trailing empty" — prevents future Output-node-hook changes from drifting out of the undo contract.
  • Save-point markers. No "document saved at this point" notion yet. NodeDesigner / MatterCAD use it to drive the "modified" titlebar indicator and the "Discard unsaved changes?" prompt on close. Add an UndoBuffer::mark_saved() + is_at_saved_point() upstream in agg-gui.
  • Batch grouping API. BatchCmd exists for ad-hoc grouping (used by the Replace-Connection path) but agg-gui's UndoBuffer has no begin_batch() / end_batch() transactional API for callers that want to accumulate multiple commands during a single user action. Add when a flow needs it (e.g. paste-multiple-nodes).
  • History size + memory profile. Default cap is 200 commands. For projects with large meshes (each AddNodeCmd on a MeshNode holds the full NodeInstance including the Geometry3d Arc<MeshGL>), an Arc-share keeps it cheap, but a Vec of 200 removed mesh nodes could cost real memory. Audit once the import + delete workflow has any traction.

7. Graph engine extensions

  • Evaluation scheduling hardening — native builds already run schedule_evaluate on a background thread (WASM stays synchronous). Missing NodeDesigner-style pieces from matter-graph-async-execution.js: debouncing, cancellation / stale-result suppression when a parameter changes mid-run, and a bounded worker model for heavy ops.
  • Component port model — required by §1.6 before components are user-facing. Treat component inputs / outputs as stable graph-boundary socket declarations with their own UIDs, labels, order, types, and persisted metadata. Parent component instances must mirror those ports reliably, and edits inside the component must propagate to existing instances without breaking saved noodles.
  • Expression evaluation in number fields — see §2 (the *OrExpression editors). Cross-cuts engine + property panel: properties become formulas referencing other properties by name ($radius * 2 + 0.5). Sandboxed evaluator with variable lookup against the node's own properties + scene globals.
  • Drivers / animated values — NodeDesigner has time-driven node parameters. Lower priority unless animation export is on the roadmap.
  • Source-side hooks — engine has on_input_connected / _disconnected / validate_input_connection. Symmetric output-side hooks not implemented. Add when a node needs to react to its consumers.
  • Source-side wildcards — JS treats output type * as "match any input". atomartist supports target-side SocketType::None wildcard only.
  • Invalidation flags — MatterCAD's InvalidateType enum (Children, Color, Image, Matrix, Mesh, Name, OutputType, Path, Properties) is finer-grained than our dirty-bit. Important for performance once node counts grow.
  • Node tags / output type — MatterCAD's PrintOutputTypes { Default, Solid, Hole } drives boolean intent. Even without printing, "this geometry is a subtractor" tagging is useful as a semantic shortcut.
  • connectByType / connectByTypeOutput — JS convenience helpers flagged in tests/node_connections.rs. Add when a UI flow demands them.

9. Decisions still pending

  • Negative-determinant transforms. Mirror node (§1.2) needs apply_transform to flip triangle winding when det(M) < 0. Add a winding-flip step in geometry/mesh3d::apply_transform guarded on the sign of the 3×3 determinant.
  • Project format evolution. .atmr zip-with-assets is the current container. Still decide whether to add a manifest / magic entry and how strict future schema migrations should be before the next breaking graph format change.
  • Field architecture. Closure-on-PortValue vs separate sample-time kernel (§1.7). Defer until the first field-consumer node lands.
  • Boolean ops + per-vertex colour propagation. The renderer's per-vertex colour path is live, but boolean nodes still emit single-colour output. Decide whether the cut-surface vertices inherit from the closest input (NodeDesigner pattern) or get a third "cut surface" colour (some CAD tools). Affects how the §3 "Per-vertex colour producers" bullet lands.
  • Dirty-state source of truth. Typed undo commands are live; ChangeTracker still exists as a snapshot-based dirty detector. Decide whether save-point tracking belongs in UndoBuffer, ChangeTracker, or a thin bridge between them.
